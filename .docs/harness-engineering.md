# Harness 实践说明

## 背景

本文档结合 OpenAI 于 2026 年 2 月 11 日发布的《[工程技术：在智能体优先的世界中利用 Codex](https://openai.com/zh-Hans-CN/index/harness-engineering/)》整理，目标不是复制文章结构，而是把其中适合 `nz-js-toolkit` 的做法落到当前仓库。

文章里对本仓库最有参考价值的点有三类：

- `AGENTS.md` 应该是短入口，不应该变成巨大的说明书
- 仓库内的 `docs/` 应该承担记录系统角色
- 规则最好能被命令、测试、lint、目录结构或脚本验证

## 在本仓库中的落地方式

### 1. 用短 `AGENTS.md` 做入口

- 根 `AGENTS.md` 只保留稳定约束、目录导航和高频命令
- 具体知识按主题下沉到 `docs/` 和包内 README
- 避免把一次性任务上下文长期塞在入口文件里

### 2. 用 `docs/` 做记录系统

当前建议至少维护三类信息：

- 架构信息：包职责、依赖方向、变更应该落在哪里
- 工作方式：如何为智能体提供稳定上下文、如何做最小验证
- 执行计划：中大型任务的目标、步骤、决策和结论

### 3. 用“渐进式披露”代替大而全提示

进入仓库时，推荐读取路径：

1. `AGENTS.md`
2. `docs/index.md`
3. 与任务直接相关的主题文档
4. 目标包的 `README.md`、`src/`、`test/`

这样可以减少上下文噪音，让智能体先拿到地图，再按需深入。

## 适合当前 monorepo 的工作回路

### 小改动

1. 确认改动属于哪个包
2. 读取该包 README 与相关源码
3. 修改最小必要文件
4. 补受影响测试或类型校验
5. 运行该包级别验证
6. 若对外行为变化，同步更新 README 或 `docs/`

### 中大型改动

1. 在 `docs/exec-plans/` 建立任务计划
2. 先拆包边界，再拆实施步骤
3. 每完成一个阶段就更新计划状态
4. 将稳定结论回写到架构文档或包文档

## 约束优先级

在这个仓库里，建议按下面顺序理解约束：

1. 代码与测试
2. 包内 README
3. `docs/` 中的结构化文档
4. 根 `AGENTS.md`
5. 临时提示词

原因很直接：越接近代码、越容易验证的内容，越适合作为智能体长期依赖的上下文。

## 建议逐步补齐的 harness 能力

结合当前仓库状态，后续可继续补的不是“大提示词”，而是更可执行的支撑结构：

- 包级检查清单模板
- `docs/exec-plans/active` 与 `completed` 中的任务沉淀
- 面向 `map-ol` 与 `crud` 的更细粒度设计文档
- 文档与代码一致性检查脚本
- 针对发布流程的 checklist

## 不建议的做法

- 把所有规范都堆进一个超长 `AGENTS.md`
- 只在聊天里约定架构，不回写仓库
- 修改行为后不更新 README / docs
- 直接编辑 `dist/` 产物替代修改源码

## 这份文档的用途

如果后续你继续实践 harness，这份文档应该不断把“有效的工作方式”沉淀为仓库资产，而不是停留在单次对话里。
